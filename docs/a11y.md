# Accessibility conventions (`ui/` + layout shells)

Cutline follows [WCAG 2.1 AA](https://www.w3.org/TR/WCAG21/) patterns where practical. These rules apply to `src/lib/components/ui/**` and layout shells under `src/lib/components/layout/**`.

## Interactive elements

- Use `<button type="button|submit|reset">` or `<a href="...">` — never a clickable `<div>` unless documented with `role`, `tabindex`, and keyboard handlers.
- Every icon-only control needs a visible label or `aria-label` (see `IconButton`).
- Search fields need `aria-label` or an associated `<label>` (see `Input`).

## Modals (`Modal`)

- `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing at the title.
- Close on **Escape**, close button, and overlay click (when `onclose` is provided).
- Move focus into the dialog on open; restore focus on close.
- Tab cycles within the dialog panel.

## Transport & media controls

- Play / pause / skip buttons use `aria-label` (wired in M2-09 / M5-11).
- Video preview exposes player state to assistive tech when controls are custom.

## Lists & navigation

- Sidebar nav uses `<nav aria-label="...">` with real links when `href` is known; inactive items stay non-interactive (`<span>`) until routes land.
- Use keyed `{#each}` with stable IDs for list items.

## Status & empty states

- `EmptyState` uses `role="status"` with a concise `aria-label`.
- Progress bars expose `role="progressbar"` with `aria-valuenow` / `aria-valuemin` / `aria-valuemax`.

## Lint enforcement

| `svelte/button-has-type` | error |
| `svelte/no-target-blank` | error |
| `svelte/no-at-html-tags` | error |
